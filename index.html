A. 連線（Jun Action Item #1）

主軸：把「連不上 / 連了又斷」止血。目標 0.1% 是長期目標，30 天交「可量測的改善 + baseline」。

1 BLE 連線止血（高頻失敗 + auto-reconnect） ✅ 30天能做
問題主題
Spark 用 BLE 連手機時，常連不上或連上後掉線；目前量測基準 BLE E2E 44 個指令裡有 31 個會 timeout（anchor 票 FWP-785）。連線根因從來沒被專門投入修過。
涉及產品
Reactor BLE 為主，連線層共用，改善外溢到其他 Spark 機種。
要做什麼
  • 先量 baseline：跑 44 指令 E2E，記錄目前成功率（這是要交給 PM 的數字依據）。
  • 挑高頻失敗的子集先修（不是一次修全部），讓成功率有可量測的提升。
  • 加 auto-reconnect：掉線後自動重連，不要逼使用者手動重開。
這條不承諾 0.1%。0.1% 要完整 BT 重構 + 跨「機型×OS」回歸，是跨季度工程。30 天只交「子集止血 + baseline」。

B. 韌體更新（Jun Action Item #2、#3）

主軸：降低「更新失敗」客訴。30 天交 updater 小修 + 一份硬體可行性評估；不在上市機上重寫更新路徑（變磚風險）。

4 更新工具小修：requiresUpdate() 防重刷 + 刷完驗證 ✅ 30天能做
問題主題
桌面更新工具 ReactorFwUpdater 裡 requiresUpdate() 永遠回傳 true → 裝置明明已是最新版，工具還是再刷一次；而且刷完不會驗證有沒有成功。（由 pg-fw-update-agent 分析挖出。）
涉及產品
ReactorFwUpdater（Gen-1 USB 更新工具，C/C++ + JUCE + dfu-util + esptool）。打到的票：STFS-491（工具起不來）、STFS-492（舊 FW 無法更新）。
要做什麼
  • 修 requiresUpdate() 邏輯：已是最新版就不要再刷。
  • 在 ReactorFwUpdater 加 post-flash verify：刷完比對確認真的寫成功。
範圍小、不碰更新架構、低風險。便宜但直接降客訴，所以排進 30 天。

5 更新流程 UX 可行性評估（unified app / web flasher） 📋 評估・不寫 code
問題主題
PM 想做「unified app（一個 app 統一更新所有 Spark 產品）」或「web flasher（瀏覽器一鍵更新）」。App / web 只是寫介面的人；更新最後是把 image 透過某通道寫進 MCU flash，這個通道與寫入方式由硬體決定，App 改不了。所以要 FW 回答「底層硬體到底支不支援、天花板在哪」。
涉及產品
全 Spark 更新線：Reactor Gen-2Spark 40/MINI/GOSpark 2/EDGE/PedalGen-1Control X / NEO
要做什麼
（FW 要給的資料）
一份評估文件，不是 code。把這幾件事填出來：
  • ① 每個產品用什麼通道更新：Reactor Gen-2＝OTA-over-WiFi（app 只觸發 enterFirmwareUpdateMode＋配 WiFi，裝置自己拉 image）；Gen-1＝USB ReactorFwUpdater；舊的 40/MINI/GO/NEO＝Python-2 BT「Actions SDK」路徑。結論點：這些根本不是同一套機制 → 「真正一包打通」先天就分裂。
  • ② web flasher 可不可行：走瀏覽器 WebUSB / WebSerial。哪些產品的 DFU 介面瀏覽器抓得到？哪些需要 driver（libwdi/WinUSB，瀏覽器裝不了）→ 直接不可行？走 BT「Actions SDK」的舊機種瀏覽器完全碰不到 → 排除。
  • ③ BLE OTA 的硬體天花板（若想無線一鍵更新）：BLE 實際吞吐 ≈ 多少 KB/s、刷一包 MB image 要幾分鐘？中途斷線會不會變磚？有沒有 atomic / 復原模式當安全網？
  • ④ 一句話結論（PM 只看這個）：例如「統一入口可行，但底層仍是 3 套機制，無法真正一包打通；web flasher 對 A/B 可行、對 C/D 因 driver/BT 不可行；BLE 一鍵更新在加安全網前不建議上市機。」
